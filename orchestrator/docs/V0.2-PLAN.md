# Sisyphus v0.2 架构演进（开发完成，待部署）

**commit 范围**：`ff13f8f..405193f`（共 9 commit）+ ttpos-ci `db24441`
**截至**：2026-04-22
**当前生产版本**：v0.1.1（还在 vm-node04 跑 REQ-997 sha；v0.2 **未部署**）

---

## 一句话

v0.2 是从 "docker run runner + 单 repo 假设 + 硬编码 lab + sonarqube 绕过 CI" 到 **"K8s Pod + PVC + 多 repo manifest + 通用 accept + 小步快跑 + 真 PR CI"** 的整体升级。sisyphus 变成纯通用编排+环境资源提供者，repo 拓扑和 accept 策略全交给 agent + manifest.yaml。

---

## 架构变化对照

| 维度 | v0.1.1 | v0.2 |
|---|---|---|
| runner 形态 | docker container on vm-node04 | **K8s Pod + PVC** in ns `sisyphus-runners` |
| 命令调用 | `aissh exec_run "docker exec ..."` | `kubectl exec runner-<req> -- ...` |
| workspace 持久化 | docker named volume | K8s PVC（生命周期绑 REQ）|
| 多 repo 拓扑 | 硬编码单 repo | `/workspace/.sisyphus/manifest.yaml`（agent 维护）|
| CI 执行 | sisyphus 自跑 ci-unit / ci-int | 移到 dev-agent 本地 + PR 触发 GHA（staging-test + pr-ci-watch 两段）|
| accept lab | 硬编码 ttpos-arch-lab | 通用：Makefile `ci-accept-env-up`/`down` + sisyphus 注入 env var |
| image tag | dev-agent 假贴 | GHA `image-publish` job 产出，写到 commit status description |
| bug 分类 | 笼统 `diagnosis:xxx` | 正交维度：**阶段标签** `bug:pre-release`/`ci`/`post-release` + `diagnosis:xxx` |
| pause/resume | 无 | `/admin/req/{id}/pause` & `/resume`（删 Pod 保 PVC / 重建 Pod）|
| rebuild workspace | 无 | `/admin/req/{id}/rebuild-workspace` |
| runner GC | 无 | 启动 + 1h 周期扫 → 清 done/escalated 过保留期 |
| agent 数量 | 9 | 12（+staging-test + pr-ci-watch + teardown 是 sisyphus 内，不是 agent）|

## 新状态机

```
INIT → ANALYZING → SPECS_RUNNING → DEV_RUNNING
                                      ↓ dev.done
                                 STAGING_TEST_RUNNING   ← 调试环境跑 unit+int（小步快跑）
                                      ↓ staging-test.pass
                                 PR_CI_RUNNING          ← PR 开了，等 GHA 全套（含 image-publish）
                                      ↓ pr-ci.pass
                                 ACCEPT_RUNNING         ← sisyphus 先 env-up，dispatch accept-agent
                                      ↓ accept.pass/fail
                                 ACCEPT_TEARING_DOWN    ← 必跑 env-down（sisyphus 直调）
                                      ↓ teardown-done.{pass,fail}
                                 ARCHIVING → DONE  或  BUGFIX_RUNNING

任意 stage 失败 → BUGFIX（分不同 bug:xxx tag）
reviewer.pass → STAGING_TEST_RUNNING（重跑小步快跑）
```

14 state × 23 event，比 v0.1.1 的 14×18 多 5 event。

## 部署前准备（打 helm upgrade 之前）

### 1. 创建 sisyphus-runners namespace 的前置 secret

```bash
# a) GitHub PAT（scope: repo + read:packages）用于 git clone / docker login ghcr.io
kubectl -n sisyphus-runners create secret generic sisyphus-runner-secrets \
  --from-literal=gh_token=ghp_xxxxx \
  --from-literal=ghcr_user=<github-user> \
  --from-literal=ghcr_token=ghp_xxxxx

# b) kubeconfig（给 runner Pod 内 agent 调 helm 起 accept lab 用）
kubectl -n sisyphus-runners create secret generic sisyphus-runner-kubeconfig \
  --from-file=config=$HOME/.kube/config
```

### 2. Runner image rebuild

S3 加了 `scripts/validate-manifest.py` 进 runner image。推 sisyphus repo 后 GHA 自动 rebuild `ghcr.io/phona/sisyphus-runner-go:main`。

### 3. ttpos-ci 配 PAT + GHCR

在 `phona/ttpos-ci` GitHub Secrets 里确认 `SOURCE_REPO_PAT` 有 `write:packages` 权限（push 镜像用）。

### 4. helm upgrade

```bash
helm -n sisyphus upgrade orch ./orchestrator/helm -f my-values.yaml
# 确认 sisyphus-runners ns 和 RBAC / secrets 都就位
kubectl -n sisyphus-runners get sa,role,rolebinding,secret
```

### 5. source repo 接入（每个）

按 `docs/sisyphus-integration.md` 做：
- `.claude/skills/{unit-test,integration-test}/`
- Makefile 加 6 个 target（`ci-lint/ci-unit-test/ci-integration-test/ci-build/ci-accept-env-up/ci-accept-env-down`）
- `.github/workflows/ci.yml` 调 `phona/ttpos-ci/.github/workflows/ci-go.yml`

## 回滚策略

v0.2 全部代码在 orchestrator image + helm chart 层。如果部署后发现严重问题：

```bash
# 1. 回退 orchestrator image 到 v0.1.1 tag
helm -n sisyphus upgrade orch ./orchestrator/helm \
  --set image.tag=<v0.1.1-sha> -f my-values.yaml

# 2. 清理 sisyphus-runners 下残留
kubectl -n sisyphus-runners delete pod --all
kubectl -n sisyphus-runners delete pvc --all

# 3. PG 里 state 可能要清 v0.2 新 state 的 REQ
# （或靠 orchestrator GC + escalate 推过终态）
```

v0.1.1 的 docker-based runner 流程不受 v0.2 helm 影响（只是 action 不会再触发 docker run），回退后旧 REQ 还能正常推进。

## 对外承诺总结

sisyphus **做**的事：
- 每 REQ 拉一个 K8s Pod + PVC 做调试环境
- 注入环境变量（REQ_ID / IMAGE_TAGS / KUBECONFIG 等）
- 编排状态机 + 分发 BKD agent 任务
- 提供 admin 运维 endpoints（pause/resume/rebuild/emit/escalate）
- GC 清过保留期的 PVC

sisyphus **不做**的事：
- 读 manifest.yaml 做业务判断（agent 管）
- 内置 helm chart 模板（repo 自己管）
- 决定 multi-repo 合并顺序（analyze/dev-agent 声明 → done-archive 执行）
- 写 contract/unit test（`.claude/skills/` 约定 → agent 实现）
