## 一句话

v0.2 是从 "docker run runner + 单 repo 假设 + 硬编码 lab + sonarqube 绕过 CI" 到 **"K8s Pod + PVC + 多 repo manifest + 通用 accept + 小步快跑 + 真 PR CI"** 的整体升级。sisyphus 变成纯通用编排+环境资源提供者，repo 拓扑和 accept 策略全交给 agent + manifest.yaml。

## 架构变化对照

| 维度 | v0.1.1 | v0.2 |

| 多 repo 拓扑 | 硬编码单 repo | `/workspace/.sisyphus/manifest.yaml`（agent 维护）|

| accept lab | 硬编码 ttpos-arch-lab | 通用：Makefile `ci-accept-env-up`/`down` + sisyphus 注入 env var |

## 新状态机

```
INIT → ANALYZING → SPECS_RUNNING → DEV_RUNNING
                                      ↓ dev.done
                                 STAGING_TEST_RUNNING   ← 调试环境跑 unit+int（小步快跑）
                                      ↓ staging-test.pass
                                 PR_CI_RUNNING          ← PR 开了，等 GHA 全套（含 image-publish）
                                      ↓ pr-ci.pass
                                 ACCEPT_RUNNING         ← sisyphus 先 env-up，dispatch accept-agent
                                      ↓ accept.pass/fail
                                 ACCEPT_TEARING_DOWN    ← 必跑 env-down（sisyphus 直调）
                                      ↓ teardown-done.{pass,fail}
                                 ARCHIVING → DONE  或  BUGFIX_RUNNING

任意 stage 失败 → BUGFIX（分不同 bug:xxx tag）
reviewer.pass → STAGING_TEST_RUNNING（重跑小步快跑）
```

14 state × 23 event，比 v0.1.1 的 14×18 多 5 event。

## 部署前准备（打 helm upgrade 之前）

### 1. 创建 sisyphus-runners namespace 下的单一 runner secret

**一个 secret 搞定**：env 注入 + kubeconfig 文件挂载都从它取。

#### 1a. Runner PAT 权限原则：最小起步，按需追加

runner 是调试环境，**所有写操作**（git push / gh pr create / gh pr merge /
openspec apply）都走 BKD agent 自己 Coder cwd 的 `GH_TOKEN`（Coder 默认有写权限）。
runner 自己用的 PAT 原则是**只读**。

**优先方案：Fine-grained PAT**（强推 — 精确到 repo 级 + 读写分离）：

| Permission | 值 | 为啥需要 |
|---|---|---|
| Repository access | 只选 sisyphus 涉及的 repo（list） | 避免全 org 泛权 |
| Contents | **Read** | git clone / git pull source repo |
| Metadata | Read（自动）| 基础 |
| Commit statuses | **Read** | pr-ci-watch agent 查 GHA CI 状态 |
| Account/Org Packages | **Read** | GHCR docker pull（org 下包要选 org 为 resource owner 且 org 开 fine-grained PAT 批准）|

如果 phona org 还没开 fine-grained PAT 批复（Settings → Personal access tokens），
同时 repo 是私有，fine-grained 这条路暂走不通。降级：

**降级方案：Classic PAT + 精确 scope**

- 如果要接入的 repo **全部公开**：`repo:status` + `public_repo` + `read:packages` 够
- 如果**任一私有**：只能选完整 `repo`（+ `read:packages`）。它确实过权，
  必须配下方 "5. Runner PAT 泄漏减灾" 所有 4 条。

**最稳路线（推荐 v0.2.1 升级）**：**Deploy Key（SSH 只读）+ read:packages 单独 PAT**
- 每个 source repo 加一把 read-only Deploy Key，runner 挂对应 SSH 私钥
- git clone 走 SSH，不经 PAT
- GHCR pull 用独立的 classic PAT `read:packages` only
- 泄漏场景：只能读单 repo 代码，**改不了任何东西**

#### 1b. 建 secret（命令）

机器账号建 PAT 后：

```bash
kubectl -n sisyphus-runners create secret generic sisyphus-runner-secrets \
  --from-literal=gh_token=<readonly-PAT> \
  --from-literal=ghcr_user=<机器账号 user name> \
  --from-literal=ghcr_token=<readonly-PAT 或独立 read:packages PAT> \
  --from-file=kubeconfig=$HOME/.kube/config
```

或 helm values（`--set-file` 读文件内容）：

```bash
helm -n sisyphus upgrade orch ./orchestrator/helm -f my-values.yaml \
  --set runner.secret.gh_token=<readonly-PAT> \
  --set runner.secret.ghcr_user=<user> \
  --set runner.secret.ghcr_token=<readonly-PAT> \
  --set-file runner.secret.kubeconfig=$HOME/.kube/config
```

#### 1c. 另一个 PAT（ttpos-ci 用，跟 runner 分开）

build + push image 的 `SOURCE_REPO_PAT`，放在 `phona/ttpos-ci` repo Secrets 里：
- scope：**`repo` + `write:packages`**
- 因为要 push image 到 ghcr.io/phona/...，必须有 write:packages
- 只 ttpos-ci 的 GHA workflow 用，不进 runner Pod
- runner PAT 泄漏不影响它，反之亦然 — 分层防御

### 2. Runner image rebuild

S3 加了 `scripts/validate-manifest.py` 进 runner image。推 sisyphus repo 后 GHA 自动 rebuild `ghcr.io/phona/sisyphus-runner-go:main`。

### 3. ttpos-ci 配 PAT + GHCR

在 `phona/ttpos-ci` GitHub Secrets 里确认 `SOURCE_REPO_PAT` 有 `write:packages` 权限（push 镜像用）。

### 4. helm upgrade

```bash
helm -n sisyphus upgrade orch ./orchestrator/helm -f my-values.yaml
# 确认 sisyphus-runners ns 和 RBAC / secrets 都就位
kubectl -n sisyphus-runners get sa,role,rolebinding,secret
```

### 5. Runner PAT 泄漏减灾（如果走 classic `repo` scope 路线**必配**）

1. **机器账号 sisyphus-bot**：别用人类个人账号挂 PAT；机器账号加到 phona org，
   repo access 只给 sisyphus 要接入的那几个
2. **所有 source repo 开 branch protection**（Settings → Branches → Add rule）：
   - main 分支 require PR + approving review + status check passing
   - **block force pushes**
   - 这样即使 PAT 泄漏，攻击者也不能 `git push --force origin main`
3. **PAT rotation ≤ 90 天**：calendar 提醒或 ExternalSecrets 自动轮换
4. **GitHub Push Protection + Secret Scanning**：repo Settings → Code security
   开起来，防 PAT 不小心 commit 进代码

Deploy Key 路线（v0.2.1）生效后，这 4 条里 1/3 还有价值，2/4 仍必须。

### 6. source repo 接入（每个）

按 `docs/sisyphus-integration.md` 做：
- `.claude/skills/{unit-test,integration-test}/`
- Makefile 加 6 个 target（`ci-lint/ci-unit-test/ci-integration-test/ci-build/ci-accept-env-up/ci-accept-env-down`）
- `.github/workflows/ci.yml` 调 `phona/ttpos-ci/.github/workflows/ci-go.yml`
- Branch protection on `main`（见 5.2）
- 预建 sisyphus labels：`sisyphus:managed` / `sisyphus:pre-release-bug` /
  `sisyphus:ci-bug` / `sisyphus:post-release-bug`（`gh label create ...`）

## 回滚策略

v0.2 全部代码在 orchestrator image + helm chart 层。如果部署后发现严重问题：

```bash
# 1. 回退 orchestrator image 到 v0.1.1 tag
helm -n sisyphus upgrade orch ./orchestrator/helm \
  --set image.tag=<v0.1.1-sha> -f my-values.yaml

# 2. 清理 sisyphus-runners 下残留
kubectl -n sisyphus-runners delete pod --all
kubectl -n sisyphus-runners delete pvc --all

# 3. PG 里 state 可能要清 v0.2 新 state 的 REQ
# （或靠 orchestrator GC + escalate 推过终态）
```

v0.1.1 的 docker-based runner 流程不受 v0.2 helm 影响（只是 action 不会再触发 docker run），回退后旧 REQ 还能正常推进。

## 对外承诺总结

sisyphus **做**的事：
- 每 REQ 拉一个 K8s Pod + PVC 做调试环境
- 注入环境变量（REQ_ID / IMAGE_TAGS / KUBECONFIG 等）
- 编排状态机 + 分发 BKD agent 任务
- 提供 admin 运维 endpoints（pause/resume/rebuild/emit/escalate）
- GC 清过保留期的 PVC

sisyphus **不做**的事：
- 读 manifest.yaml 做业务判断（agent 管）
- 内置 helm chart 模板（repo 自己管）
- 决定 multi-repo 合并顺序（analyze/dev-agent 声明 → done-archive 执行）
- 写 contract/unit test（`.claude/skills/` 约定 → agent 实现）
