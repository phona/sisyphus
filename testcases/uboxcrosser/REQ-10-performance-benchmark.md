| Field | Value |

## Overview

定义 sisyphus 的性能评估方法。性能不是"能不能做"，而是"做得多快、多省、多好"。需要人类基准对照。

---

## 1. 性能指标定义

### 1.1 效率指标

| 指标 | 定义 | 采集方式 |
|------|------|---------|
| **端到端耗时** | 从需求下发到 PR 合并的总时间 | n8n 时间戳 |
| **有效编码时间** | 去掉等待 CI、等待审批的纯编码时间 | AI session 时长 |
| **首次提交时间** | 从需求下发到第一�� push 代码 | git log |
| **CI 等待时间** | 等待 CI 执行的累计时间 | GitHub Actions |
| **修复轮次** | CI 失败 → 修复 → 再跑的循环次数 | n8n 计数 |

### 1.2 成本指标

| 指标 | 定义 | 采集方式 |
|------|------|---------|
| **Token 消耗** | AI 调用的总 input+output token | vibe-kanban API |
| **Token 成本（美元）** | token × 单价 | 计算 |
| **CI 分钟数** | GitHub Actions 消耗的运行分钟 | GitHub API |
| **人工干预次数** | 需要人介入的次数 | 手动记录 |
| **人工干预时间** | 人介入的累计时间 | 手动记录 |

### 1.3 质量指标

| 指标 | 定义 | 采集方式 |
|------|------|---------|
| **首次 CI 通过率** | 第一次 push 就 CI 绿的概率 | GitHub Actions |
| **Lint 干净度** | 最终提交的 lint warning 数 | golangci-lint |
| **测试覆盖率** | 新增代码的覆盖率 | SonarQube |
| **代码膨胀率** | 实际改动行数 / 最小必要行数 | `git diff --stat` vs 人工评估 |
| **AC 通过率** | 通过的验收标准数 / 总验收标准数 | 人工验收 |

## 3. 性能比值

核心性能指标用**比值**表示，大于 1 表示 sisyphus 优于人类：

```
速度比 = 人类耗时 / sisyphus 耗时
成本比 = 人类成本(工资×时间) / sisyphus 成本(token+CI)
质量比 = sisyphus AC通过率 / 人类 AC通过率
效率比 = (速度比 + 成本比 + 质量比) / 3
```

### 成本换算

```
人类成本 = 日薪 × 天数
sisyphus 成本 = token费 + CI分钟�� + 人工干预时间×日薪/8h

假设：
  - 中级开发者日薪: $300 (参考值)
  - Claude Opus token: $15/M input, $75/M output
  - GitHub Actions: $0.008/min
```

## 4. 吞吐量测试

单个任务的效率只是一面。还要测**并发吞吐量**：

### 4.1 串行吞吐

依次执行 REQ-01 → 06 → 02 → 03 → 04 → 05，记录总耗时。

### 4.2 并行吞吐

同时下发 REQ-02 + REQ-03（可并行的任务对），对比：
- 串行执行总耗时 vs 并行执行总耗时
- 并行加速比 = 串行耗时 / 并行耗时
- 理想值 ≈ 2（完美并行），实际受资源争用影响

### 4.3 批量吞吐

一次性下发 REQ-01~06（6 个任务），测 sisyphus 的：
- 总完成时间
- 调度效率（空闲等待时间占比）
- 资源利用率（并发 namespace 使用率）

---

## 5. 稳定性测试

同一个 REQ 执行 3 次，观察：

| 指标 | 计算 |
|------|------|
| 耗时方差 | 3 次耗时的标准差 |
| 结果一致性 | AC 通过数是否一致 |
| 路径一致性 | 修改的文件集合是否一致 |
| 代码风格一致性 | 3 次产出的代码是否风格统一 |

高方差 = 不稳定，结果靠运气
低方差 = 可预测，可信赖
